最近はClaudeやChatGPTを使わない人はいませんよね。でも、それを**ネットも料金もなしに、自分のノートで直接**動かせるとしたら? 「そんなのGPUがずらっと並んだサーバー室がいるんじゃ?」と思いますが、結論から言うと**家にあるゲーミングノート1台で、一円もかけずに**できます。前回はComfyUIで「絵を描くローカルサーバー」を作りましたが、今回は`llama.cpp`で**「話す・書くLLM」**を自分のGPUに載せてみた実践記録です。

## なぜわざわざローカルで動かすのか?

正直、賢さだけならクラウドAIが上です。それでもローカルで動かす理由がはっきりあります。

いちばん大きいのは**トークンを無制限に使える**こと。クラウドAPIは入出力のトークン一つひとつに課金されます。文書を数百件要約したり、ログを数万行分類したり、一晩中何かを回したりすると、料金がじわじわと恐ろしく膨らみます。ローカルは電気代以外は0円なので、「これトークンもったいないな…」とためらう必要がありません。プロンプトを好きなだけ長く書けるし、納得いくまで何度でも回せます。

さらにこんな利点も付いてきます。

- **レート制限がありません。** APIは分あたりのリクエスト数に上限があり、大量処理では待たされ続けますが、自分のGPUは完全に自分のものなので行列とは無縁です。
- **データが外に出ません。** 社内文書、個人情報、まだ公開していないコードなど、外にアップしづらいものを扱うときにこれが決め手になります。そもそも送信しないので漏れようがありません。
- **ネットが要りません。** 飛行機の中でも、ネットが遮断された閉域網でもそのまま動きます。
- **モデルが突然消えません。** クラウドモデルはある日旧版が終了したりしますが、自分のディスクに落としたファイルは永遠にそのバージョンのままです。

一言でいえば**「賢さはクラウド、自由度・コスト・プライバシーはローカル」**。機密データを扱うほど、同じ作業を大量に繰り返すほど、ローカルが輝きます。

## ローカルLLM、何で動かす? — 候補3つ

いくつかやり方がありますが、3つを比べて選びました。

| ツール | 長所 | 惜しい点 |
|------|------|-----------|
| **Ollama** | コマンド1行ですぐ起動、いちばん簡単 | 中身は結局llama.cpp、細かい調整には一枚挟まる |
| **LM Studio** | きれいなGUI、数クリック | プログラムから呼んで自動化するには重い |
| **llama.cpp ← 選択** | **エンジンそのもの**、VRAMの絞り出しが最強、OpenAI互換API | コマンドに少し慣れる必要あり |

実は上のOllamaやLM Studioが中で実際に動かしている**エンジンこそがllama.cpp**です。GPUに何層載せるかまで手で調整できるので8GBノートで絞り出すのに最適で、付属の`llama-server`が**OpenAIとまったく同じ形式のAPI**を開いてくれるので、どんなアプリもきれいに繋がります。

## 全体像 — 仕組みはシンプル

原理はComfyUIのときと同じです。重い計算はローカルGPUがこなし、外のプログラムはAPIで呼ぶだけ。

```
ユーザー/アプリ: 「この段落を要約して」
   ↓
llama-server (ローカル, http://127.0.0.1:8080/v1)  ← OpenAI互換API
   ↓
GGUFモデルがRTX 4060でトークンを一つずつ生成
   ↓
回答がストリーミングで流れ出す
```

覚える単語は2つだけ。**llama.cpp**はC/C++で書かれたLLM推論エンジンで、GPUが非力でもモデルをCPUとGPUに分けて載せて動かします。**GGUF**はllama.cpp専用のモデル形式で、元のモデルを**量子化**して容量とVRAM使用量を大きく削ったファイルです。

## 自分のノートでいける? — まずスペック確認

まず可能かどうか。前回のComfyUI記事と同じHP Victus 16ノートです。

| 項目 | 必要スペック (7〜8Bモデル) | 私のノート | 判定 |
|---|---|---|---|
| GPU VRAM | 6GB以上 (Q4基準) | RTX 4060 Laptop 8GB | 通過 |
| RAM | 16GB | 16GB | 通過 |
| ディスク | モデルあたり約5GB | 余裕あり | 通過 |

VRAM 8GBなら**7〜8B級モデルをQ4に量子化すれば丸ごとGPUに載ります。** ここがいちばん速い領域です。13〜14B級からは一部をRAMに逃がす必要があって速度がガクッと落ち、30B以上はこのノートには少し欲張りです。

## モデル選び — 「量子化」が半分です

同じモデルでも**量子化(quantization)**をどうするかで容量・速度・品質が変わります。ここで最初の失敗をしました。「どうせなら高品質」とQ8を落としたら、**VRAMが足りずモデルが半分しかGPUに載らない**事態に。結局8GBの現実的な正解は**`Q4_K_M`**でした。

| 量子化 | 7Bモデル容量 | 特徴 |
|---|---|---|
| Q8_0 | 約7.5GB | 元に最も近いが8GBには重い |
| **Q4_K_M** | **約4.5GB** | **速度と品質のバランス、ローカルの事実上の標準** |
| Q3_K_M | 約3.5GB | より軽い代わりに品質低下がそろそろ見えてくる |

モデルは7〜8B級のinstruct系(例: Qwen2.5-7B-Instruct、Llama-3.1-8B-Instruct など)なら無難です。GGUFファイルは通常、Hugging Faceの`bartowski`のような配布元から量子化別に入手できます。

## インストール (Windows 11 基準)

### 1. llama.cppのバイナリを入手

ソースから直接ビルドもできますが、いちばん速いのは**ビルド済みのCUDAバイナリ**を落とすこと。[GitHubリリース](https://github.com/ggml-org/llama.cpp/releases)から`llama-bxxxx-bin-win-cuda-x64.zip`のようなファイルを落として`C:\llama`に展開すれば完了です。

### 2. GGUFモデルをダウンロード

```powershell
# 7B級 Q4_K_M モデル (約4.5GB) → C:\llama\models\ に保存
curl.exe -L -o C:\llama\models\model-q4_k_m.gguf ^
  "https://huggingface.co/bartowski/Qwen2.5-7B-Instruct-GGUF/resolve/main/Qwen2.5-7B-Instruct-Q4_K_M.gguf"
```

### 3. サーバーを起動

ここで覚えるのはたった1つ。**`-ngl`** — GPUに載せる層の数です。`-ngl 99`は「載せられるだけ載せろ」の意味で、7〜8B Q4モデルは8GBに丸ごと入ります。

```powershell
C:\llama\llama-server.exe -m C:\llama\models\model-q4_k_m.gguf -ngl 99 -c 8192 --port 8080
```

- `-ngl 99` : 全層をGPUに載せる (VRAMが足りなければ数字を下げて一部だけ)
- `-c 8192` : コンテキスト長8kトークン
- `--port 8080` : サーバーポート

### 4. ちゃんと動くか確認

ブラウザで`http://127.0.0.1:8080`を開くとチャットUIがすぐ出ます。GPUを本当に使っているかは、起動時ログの`offloaded 29/29 layers to GPU`のような行で確認します。この数字が満杯なら成功です。

### よくある落とし穴3つ (先に教えます)

インストール自体は簡単ですが、みんなちょうどこの3か所でつまずきます。

- **落とし穴1 — CPUビルドを落としてしまう。** リリースにはファイルが複数あるので、うっかり`win-cpu`ビルドを落とすとGPUが遊んで回答がのろのろに。**NVIDIAなら必ず`cuda`ビルド**を。(AMDは`vulkan`ビルド)
- **落とし穴2 — `-ngl`を付け忘れる。** これを省くとモデルが全部CPUで動いて激遅になります。「GPUあるのに何でこんなに遅いの?」と思ったら十中八九これです。
- **落とし穴3 — 欲張って大きいモデル・高い量子化。** 上で味わったあの失敗です。VRAMを超えると速度が急落します。8GBなら7〜8B Q4が正解。

## アプリと繋ぐ — OpenAI互換API

`llama-server`が本当に便利なのは、**OpenAIとまったく同じ形式のAPI**を開いてくれる点です。OpenAIを使っていたコードで、**アドレスをローカルに変えるだけ**でそのまま動きます。

```python
from openai import OpenAI
client = OpenAI(base_url="http://127.0.0.1:8080/v1", api_key="local")

resp = client.chat.completions.create(
    model="local",
    messages=[{"role": "user", "content": "こんにちは、自己紹介して"}],
)
print(resp.choices[0].message.content)
```

この1行で、文書要約・コード補完・翻訳を**オフライン・無課金**で回す自分だけのバックエンドが完成します。

## どのくらい速い?

- 7〜8B Q4_K_Mを全層GPUに載せた場合で**毎秒40〜60トークンほど**。目で追いきれないくらい速いです。
- 最初の応答までは、モデルをVRAMに載せる時間を含めて数秒。
- コンテキストを長く(例: 16k)すると VRAMが足りなくなることがありますが、そのときは`-ngl`の数字を下げて一部をRAMに逃がせばOK。少し遅くなる代わりに動きます。

## 正直な結論 — どこまでできて、どこから無理か

- **できる:** 無料で、ネットなしで、自分のノートでかなり使えるLLMが動きます。要約・翻訳・分類・コーディング補助のような実務なら7〜8Bで十分。何より**トークンを気にせず思い切り**回せるのが大きい。
- **限界:** どんなに良いローカルモデルでも、最上位のクラウドモデル(Claudeなど)ほど賢くはありません。複雑な推論や長いコード設計はまだクラウドが上です。
- **現実的な組み合わせ:** なので私は**機密データ・大量の繰り返し作業はローカル、難しい問題はクラウド**と使い分けるのが一番だと思います。両者は競争ではなく役割分担です。

お金をかけずノート1台でここまでできること自体が、AIのハードルがどれだけ下がったかを物語っている気がします。次は、このローカルLLMをClaude Codeのようなツールに繋いで、**完全オフラインのコーディング相棒**を作ってみるつもりです。
